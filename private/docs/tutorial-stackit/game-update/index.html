Game Update

With loop structure and data structures defined for the game, we now have to use our loop's structure to update our data, bringing one game state to the next.

Update Structure#

Here's a general outline for the structure of our update function:

  • Get user input
  • Handle case that user wants to stack the platform
  • Move the platform
  • Check game states

User Input#

First thing's first, users interact with Stack It through the joystick. Without the user input, nothing interesting happens in the game.

So, let's begin working on void updateGame() by writing code to determine the joystick's button state during a game tick:

stackit.ino 
void updateGame() {    // Check if joystick is clicked    bool clicked = !digitalRead(CLICK_PIN);}

The above code checks if the joystick button is currently clicked and then stores the answer in the variable clicked.

Handle Stack#

With the joystick button read in, we can decide if the user wants to stack the platform on the tower or not.

If they do, we need to handle the case immediately. It's a bit of a complicated situation, so I'll drop the full code segment off here, and we can dive into the specifics one by one:

stackit.ino 
void updateGame() {  ... // EXISTING CODE
  // Handle stacking event    if (clicked) {        if (platform.layer == 0) {            for (int i = 0; i < platform.size; i++) {                int index = platform.layer * WIDTH + platform.x + i;                tower[index] = true;            }        } else {          int newSize = platform.size;          for (int i = 0; i < platform.size; i++) {              int platformI = platform.layer * WIDTH + platform.x + i;              int belowI = (platform.layer - 1) * WIDTH + platform.x + i;              if (tower[belowI]) {                  tower[platformI] = true;              } else {                  newSize--;              }          }          platform.size = newSize;        }        platform.layer++; // Increment the layer of the platform since the user clicked    }

Layer 0 Platform#

The first case is that the platform is on layer 0. This implies there is nothing below the platform in the tower, so we don't need to check beneath it to see if the player's stack was successful.

So, for this case we simply need to set the tower array to 'true' at the appropriate indexes and then increment the platform's layer (done at the end).

We can see that the loop in the case accomplishes exactly this using the row-column to index formula.

stackit.ino 
for (int i = 0; i < platform.size; i++) { // Loop through all of the platforms blocks    int index = platform.layer * WIDTH + platform.x + i; // row-column index formula, add i to adjust for platform block.    tower[index] = true; // Set the tower block at the platform position as active}

Layer 1+ Platform#

In other cases, the success of the player's stacking depends directly on the blocks in the tower below the platform. So, we need to check how many blocks the platform is landing on top of.

The number of blocks that the platform is on top of when the user clicks the joystick button becomes the number of blocks for future platforms. Additionally, the tower grows only by the platform blocks that are over existing blocks in the tower.

The case discussed is handled by the code below:

stackit.ino 
int newSize = platform.size; // This variable will decide the height of the platform when it moves to the next layerfor (int i = 0; i < platform.size; i++) { // Loop through all of the platforms blocks    int platformI = platform.layer * WIDTH + platform.x + i; // row-column index formula, add i to adjust for platform block.    int belowI = (platform.layer - 1) * WIDTH + platform.x + i; // row-column index formula for the tower block below the platform block.    if (tower[belowI]) { // In this case, there is an active tower block below the platform block        tower[platformI] = true; // Add the platform block to the tower's active blocks    } else { // In this case, there is no tower block below the platform        newSize--; // Platform loses size    }}platform.size = newSize; // Update the size of the platform object

Collision Detection#

Collision detection in Stack It is fairly simple, as we only have one fairly trivial case to worry about.

Out of Bounds#

In Stack It, we just need to make sure that the Platform will not move outside the game bounds of the 16x16 grid. If it will, we just have to change its direction before it does so.

We can accomplish this conditional check and appropriate action with the following code:

stackit.ino 
void updateGame() {    ... // EXISTING CODE
    // Collision detection    int newX = platform.x + platform.dir; // Calculate new x position of platform after movement    if (newX < 0 || newX >= (WIDTH - platform.size + 1)) {        platform.dir *= -1; // Change platform direction    }}

Moving the Platform#

Now that we've handled the out of bounds collision event, we just need to update the x position of the platform using our guaranteed safe platform.dir:

stackit.ino 
void updateGame() {    ... // EXISTING CODE
    // Move platform    platform.x += platform.dir;}

Check Game States#

At the end of the update process, we have to check our game state cases to determine if any special actions need to be performed. In particular, we need to decide if the player has won or lost the game.

To accomplish this, we need to break down what it specifically means for each case to be true.

Losing#

For a player to have lost, they must have missed all platform blocks when they clicked the joystick button. From the perspective of our location in the update process, we can think of this as the platform having a size of 0, as in Handle Stack we decremented the size of the platform for each missed block.

The code for the losing case thus looks like this:

stackit.ino 
void updateGame() {    ... // EXISTING CODE
    // Check losing game state    if (platform.size == 0) { // Loser!        renderBlink(0xff0000); // Make screen flash red        initGame(); // Reset game to start        return; // Stop execution of update function    }
note

We assume the existence of a function called renderBlink(hex). We will implement this feature in Render Game.

Winning#

For a player to have won, they must have successfully stacked the tower all the way to the top of the grid. In other words, the platform must have moved on to layer 16.

Thus, to check the winning case we just check if the platform is on layer 16:

stackit.ino 
void updateGame() {    ... // EXISTING CODE
    // Check winning game state    if (platform.layer == 16) { // Winner!        renderBlink(0x00ff00); // Make screen flash green        initGame(); // Reset game to start        return; // Stop execution of update function    }

Tick Rate#

Finally, let's revisit the tick rate we talked about in Stack It Setup

In Stack It, the speed of the platform increases the higher up you get. To mimic this behavior, we can subtract the current layer times a weight by an arbitrary base tick time:

stackit.ino 
void loop() {    // Update game    updateGame();
    // Render    renderGame();
    // Set tick rate    delay(350 - ((float)platform.layer / 15.0) * 250);}

In this case, the base tick rate is 350 milliseconds. Every layer that the platform increases by increases the tick rate by ~17 milliseconds.

tip

The tick rate increase here is totally arbitrary and was determined based on trial and error for what felt right. If you want to make the game harder or easier, mess with the tick rate equation and see what happens!
