Game Update

With loop structure and data structures defined for the game, we now have to use our loop's structure to update our data, bringing one game state to the next.

Update Structure#

Here's a general outline for the structure of our update function:

  • Get user input
  • Handle paddle movement
  • Check for ball collisions & handle cases
  • Move the ball
  • Move the AI paddle
  • Check gamestates

User Input#

First thing's first, users interact with Pong through the joystick. Without the user input, nothing interesting happens in the game.

In pong, the user input we care about is just the Joystick's y-axis position. This is because the only thing the user can control in Pong is if the joystick is going to move up or down vertically on the screen.

So, let's begin working on void updateGame() by writing code to determine the joystick's y-position during a game tick:

pong.ino 
void updateGame() {  // Read joystick values  int yValue = analogRead(joyY);  int yMap = map(yValue, 0 , 1023, -10, 10);}

The above code reads in the Y value from the joystick, and then maps the raw analog signal rage (0, 4095) to a more reasonable (-10, 10) range in the variable yMap.

Handle Player Paddle Movement#

With the Y-Axis value read in, we can handle moving the player's paddle.

In particular, we should check if the Joystick's Y-Axis is offset by some minimum threshold. If it is, we should then determine the direction the player wants the paddle to move. During this process we will also wrap in out-of-bounds collision detection, as it ends up being natural to check for:

stackit.ino 
void updateGame() {  ... // EXISTING CODE
  // Handle player paddle movement logic  if (abs(yMap) > 2) { // Ensure the magnitude of joystick y-axis offset is above threshold of 2      if (yMap > 0) { // Check if the joystick is pointing up          if (playerPaddle.y + 1 <= (HEIGHT - playerPaddle.size)) // Check if the paddle will go out of bounds              playerPaddle.y++; // Move paddle up      } else { // Joystick is pointing down          if (playerPaddle.y - 1 >= 0) // Check if paddle will go out of bounds              playerPaddle.y--; // Move paddle down      }  }

As you can see in the code, if we encounter an out of bounds event we just don't update to position of the paddle, leaving it where it is.

Ball Collisions#

The ball has quite a few collision cases. Namely, the following are of interest:

  • Out of bounds on horizontally on the left (AI Scores)
  • Out of bounds on horizontally on the right (Player Scores)
  • Out of bounds on vertically (Ball bounces off)
  • Collides with paddle (Ball bounces off of paddle)
pong.ino 
void updateGame() {    ... // EXISTING CODE
    // Update ball    int ballColX = round(ball.x + ball.xVel); // Calculate new ball x position    int ballColY = round(ball.y + ball.yVel); // Calculate new ball y position    if (ballColX < 0) { // Ball out of bounds left        aiPaddle.score++; // Add to player's score        newRound(); // Start new round        return; // Stop executing the updateGame function    }    if (ballColX >= WIDTH) { // Ball out of bounds right        playerPaddle.score++; // Add to player's score        newRound(); // Start new round        return; // Stop executing the updateGame function    }    if (ballColY < 0 || ballColY >= HEIGHT) { // Ball is out of bounds vertically        ball.yVel *= -1; // Reflect the ball off the wall by inverting y-velocity    }    // Ball hits player's paddle    if ((ballColX == playerPaddle.x) && ((ballColY >= playerPaddle.y) && (ballColY < playerPaddle.y + playerPaddle.size))) {        ball.xVel *= -1; // Reflect ball off of paddle by inverting x-velocity    }    // Ball hits AI's paddle    if ((ballColX == aiPaddle.x) && ((ballColY >= aiPaddle.y) && (ballColY < aiPaddle.y + aiPaddle.size))) {        ball.xVel *= -1; // Reflect ball off of paddle by inverting x-velocity    }}

Move the Ball#

Now that we've updated the velocity of the ball based on possible collision events (and stopped updating with return if either player has scored), we can simply add the velocity to the ball's position to get the updated ball position.

pong.ino 
void updateGame() {    ... // EXISTING CODE
    // Move ball    ball.x += ball.xVel;    ball.y += ball.yVel;}

AI Paddle#

Now comes the interesting part: making an AI paddle. There's a LOT of different ways to go about this. Some approaches to paddle AI can be highly sophisticated. Others can be very simple with a clever idea.

A well programmed AI should be good at the game, but not so good that a player can never beat it. Thus, the goal of an AI programmer should be to find the sweet spot in creating an AI that fits the difficultly level of the game they're aiming for.

In this tutorial, we'll try to make the simplest "medium" difficulty AI possible. To start, let's create a new function void aiUpdate() and put it above void updateGame():

pong.ino 
void aiUpdate() {}

The update procedure behind our AI will be this:

  • Generate a random number between 0-4. If the number is 4, continue. Otherwise don't move the AI paddle.
  • Check if the paddle's middle is above or below the ball's current position.
  • If above, move the paddle down. Otherwise move the paddle up.

Extremely simple! Here's the code putting that to action:

pong.ino 
void aiUpdate() {  if ((rand() % 5) % 4 == 0) {        if ((aiPaddle.y + aiPaddle.size / 2) > ball.y) {            if (aiPaddle.y - 1 >= 0)                aiPaddle.y--;        } else {            if (aiPaddle.y + 1 <= (HEIGHT - playerPaddle.size))                aiPaddle.y++;        }    }}
tip

We can even change the "difficulty" of the AI by changing the random number generation step to include more numbers (say, 0-10, leaving a 10% chance of moving the paddle). Try messing with it and see what happens!

Now we can just go ahead and call this function from our gameUpdate function to update the AI paddles position each update tick:

pong.ino 
void updateGame() {  ... // EXISTING CODE
  // Update AI  aiUpdate();}

Game States#

At the end of the update process, we have to check our game state cases to determine if any special actions need to be performed. In particular, we need to decide if the player has won or lost the game.

To accomplish this, we need to break down what it specifically means for each case to be true.

Winning#

For a player to have won, they just need to have scored WINNING_SCORE points.

We'll just have the screen flash green when the player wins and then reset the game:

pong.ino 
void updateGame() {    ... // EXISTING CODE
    // Check winning gamestate    if (playerPaddle.score == WINNING_SCORE) {        renderBlink(0x00ff00); // Blink green        initGame(); // Reset game        return; // Stop updating    }
note

We assume the existence of a function called renderBlink(hex). We will implement this feature in Render Game.

Losing#

For a player to have lost, the AI needs to have scored WINNING_SCORE points.

Similar to the winning case, we'll just have the screen flash red when the player loses and then reset the game:

pong.ino 
void updateGame() {    ... // EXISTING CODE
    // Check losing game state    if (aiPaddle.score == WINNING_SCORE) {        renderBlink(0xff0000); // Blink red        initGame(); // Reset game        return; // Stop updating    }
